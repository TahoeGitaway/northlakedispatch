Security Overview

North Lake Dispatch — how your data and operations are protected

Admin Only
✅
Platform security summary

North Lake Dispatch is a private, login-gated operations platform. All sensitive data — guest reservations, property addresses, API credentials — is handled server-side and never exposed to the browser. Access is controlled by role-based authentication, and the platform is hosted on Railway with environment-variable secrets management.

🔐

Authentication & Access Control

Implemented
✓ All routes require login. Every page and API endpoint is protected with @login_required. Unauthenticated requests are redirected to the login page — no data is served to anonymous users.
✓ Role-based access. Admin-only pages (user management, knowledge base, AI chat, security overview, PRI check, CSV upload) require @admin_required on top of login. A logged-in non-admin user cannot reach any admin endpoint.
✓ Passwords are hashed. User passwords are stored using Werkzeug's generate_password_hash (PBKDF2-SHA256). Plain-text passwords are never stored.
✓ Session management. Flask-Login manages sessions with a cryptographically signed cookie (HMAC using the server's secret key). Sessions expire after 8 hours.
✓ This page is double-gated. Access to this security overview requires both admin login and a separate PIN, so it cannot be stumbled upon by any admin user.
🔑

API Credentials & Secrets

Implemented
✓ All API keys are environment variables. Anthropic, Breezeway, Google Maps, and SendGrid credentials are stored exclusively as Railway environment variables — never in source code or committed files.
✓ Keys are never sent to the browser. API credentials are read server-side at request time. No credential ever appears in HTML, JavaScript, or API responses.
✓ Breezeway JWT tokens are short-lived. The app requests a fresh Breezeway access token on demand and uses it only server-side. Tokens are not cached to disk or returned to clients.
🏠

Guest & Reservation Data

Implemented
✓ Breezeway data stays server-side. Reservation details fetched from Breezeway are used to build AI prompts and route context on the server. Raw reservation objects are never sent to the browser.
✓ AI system prompts are not returned to the client. The chatbot endpoint previously returned the full system prompt (containing Breezeway context) in the JSON response. This has been removed — only the AI reply is returned.
✓ Debug endpoints are admin-gated. The /briefing/debug-reservations and /briefing/debug-properties endpoints that return raw Breezeway data require admin role — not just login.
✓ Server-side reservation filtering. Even if Breezeway's date-range API returns records outside the requested window, the app validates and filters all reservation dates server-side before processing.
🗺️

Maps & Third-Party Services

Implemented
✓ Map tiles require no API key. All maps use CartoDB Voyager tiles, which are served anonymously over HTTPS with no user-identifiable information transmitted.
✓ Google Maps API key is server-side only. The Google Maps Distance Matrix API (used for route optimization) is called from the server. The key is never embedded in frontend JavaScript.
✓ AI API calls are server-side. All calls to Anthropic (Claude) are made from the Flask backend. The Anthropic API key never touches the browser.
🗄️

Database & Infrastructure

Implemented
✓ PostgreSQL on Railway. The database is a managed Railway PostgreSQL instance, not publicly accessible. The connection URL is an environment variable.
✓ Parameterized queries. All database queries use %s parameterization via psycopg2 — user input is never interpolated directly into SQL strings, preventing SQL injection.
✓ Hosted on Railway. The app is deployed on Railway's isolated container infrastructure with HTTPS enforced. No local ports are exposed. All traffic is encrypted in transit.
⚠️

Planned Improvements

Roadmap
○ CSRF protection. POST endpoints currently rely on session authentication alone. Adding Flask-WTF CSRF tokens would prevent cross-site request forgery attacks from malicious third-party sites.
○ HTTP security headers. Adding X-Frame-Options, X-Content-Type-Options, and Strict-Transport-Security headers would harden against clickjacking and MIME-sniffing attacks.
○ Guest PII stripping. Breezeway reservation objects are cached in server memory with all fields intact. Stripping guest names and contact info before caching would limit exposure if memory were somehow accessed.
○ Session timeout reduction. Sessions currently last 8 hours. Reducing to 2–4 hours would limit the window of exposure on shared or unattended workstations.
